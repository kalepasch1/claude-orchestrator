  prompt: |
    runner/resource_governor.py gates lane admission on `ram_free_gb()` derived from vm_stat
    "Pages free". On macOS that number is misleading — the OS keeps most RAM in file cache and
    purgeable pools, so "free" is chronically near-zero even with plenty of reclaimable memory.
    Consequence observed 2026-07-09: the governor swings between throttle->1 (falsely starved at
    4.8GB "free") and over-admitting 13 lanes until real pressure hits 0.3GB and the runner wedges.
    Fix: compute an ACCURATE available-memory signal on macOS:
    - Prefer `memory_pressure` (parse "System-wide memory free percentage" / pressure level
      normal|warn|critical) as the PRIMARY admission signal; map warn->hold new claims,
      critical->throttle to 1.
    - For the GB math, count available = free + inactive + purgeable + speculative + file-backed
      reclaimable (from `vm_stat` page buckets), NOT just Pages-free. Fall back to the old
      Pages-free path on non-macOS or if parsing fails (keep it cross-platform + fail-soft).
    - Keep RAM_FLOOR_GB / PER_TASK_GB semantics but apply them to the corrected available number.
    - Add a hard ceiling so RUNNING can never exceed MAX_PARALLEL_CEILING even across a wedged-
      runner transition (the 13>12 anomaly): the runner must also enforce the throttle at CLAIM
      time, re-reading the throttle file every claim, not once per loop.
    Unit tests (runner/tests/test_governor_mem.py): synthetic vm_stat + memory_pressure fixtures
    for healthy/warn/critical; assert admission decisions; assert Pages-free fallback path;
    assert ceiling never exceeded. Do not change behavior on Linux runners.
