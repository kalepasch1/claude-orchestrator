- id: intake-semantic-dedup
  title: Flag intake tasks that duplicate an existing capability/module before spawning
  material: no
  model: sonnet
  depends: []
  proof: "python3 -m pytest runner/tests/test_intake_dedup.py -q exits 0 AND python3 -m pytest runner/tests -q exits 0"
  prompt: |
    Slugs are deduped but CONCEPTS are not — the same engine got rebuilt many times across sessions.
    Add runner/intake_dedup.py with a PURE function candidate_matches(prompt, existing) -> list of
    (ref, score) that ranks a new task prompt against (a) the capability registry (capability.list or
    db.select on the capabilities table) and (b) in-flight/queued task slugs+prompts (db.select tasks
    where state in QUEUED/RUNNING). Use capability.py's existing embedding dedup path when EMBED_PROVIDER
    is set, else a cheap token-overlap fallback (NO network in the unit test). Wire a hook in
    intake_watcher.py at insert time: when the top match >= INTAKE_DEDUP_THRESHOLD, DO NOT block —
    annotate the task (append "POSSIBLE-DUP: <ref> (score)" to its prompt/notes) and link it, so the
    agent reuses instead of rebuilding. Fail-open on any error. Unit-test candidate_matches on synthetic
    fixtures (a near-dup scores high, an unrelated task scores low) in test_intake_dedup.py. Keep suite green.
