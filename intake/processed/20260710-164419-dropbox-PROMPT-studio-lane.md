## DESIGN

Operator improvements are normal orchestrator tasks (full stack: prompt assembly, knowledge injection, bandit vendor/model routing, verify gates) that integrate to a per-project `design` branch — never master, never prod. The feedback loop is SECONDS (warm dev servers + hot reload), not deploy round-trips. Brand kits uploaded by designers become design tokens the whole fleet uses. Scout bots continuously propose cutting-edge UI/UX improvements; persona QA bots test variants against target-market groups so choices are demand-guided. Prod deploys from master only, via explicit `promote()`.

## PRINCIPLE 0 — ADDITIVE + OPERATOR-STEERABLE, ALWAYS

Everything in this build is ADDITIVE: no existing lane, gate, job, or workflow is removed or behavior-changed for non-studio work. And every automated decision in this system — routing choice, variant count, persona weighting, scout angle, auto-apply threshold, brand-token mapping, batch composition — must be overridable by the operator in plain language from the studio prompt box (e.g., "use gemini for this one", "only 1 variant", "ignore the SMB persona here", "don't auto-apply today"). Implement this as a lightweight instruction parser in the composer that maps operator directives onto the corresponding task fields/config for THAT task or session — prompting is the steering wheel; config flags are just the defaults it overrides.

## PART 1 — DESIGN BRANCH MECHANICS (safe, never stale, never auto-prod)

1. Per project, ensure branch `design` exists. `ORCH_STUDIO_BASE` (default `design`) in fleet_config.
2. `runner/design_sync.py` (every 15 min, fail-soft): merge master → design per project (merge, not rebase — shared long-lived branch). On conflict: file a `studio-sync-fix` task (prefer design for style files, master for logic), notify; never guess, never block the master lane.
3. Deploy guard: production Vercel deploys ONLY from master (test-asserted). `design` pushes create preview deployments only.
4. `promote(project)`: design → master through the normal merge train with ALL gates; then fast-forward design to master. Dashboard button + `operator.sh promote <project>`.

## PART 5 — PREFERENCE LEARNING → AUTO-APPLY (multiply operator attention)

20. Feed every studio signal — variant picks, rejects, promote decisions, retry-with-stronger-model events — into `owner_decision_model.py` as labeled preferences (diff features: token deltas, component types, density, palette direction).
21. Auto-apply never touches master; promote() stays explicit. Track prediction accuracy on the scoreboard; demote the threshold automatically if revert rate exceeds 10%.

## PART 6 — UX SCOUT BOTS (continuous cutting-edge improvement suggestions)

22. `runner/ux_scout.py` (periodic, generator-classed, respects drain_mode + caps): each cycle picks one project and one ANGLE from a rotating portfolio — visual design trends, interaction patterns, UX friction (navigation depth, form length, dead-ends), user-friendliness (copy clarity, empty states, error states), accessibility (WCAG contrast/keyboard/screen-reader), performance-as-UX (LCP, layout shift), mobile ergonomics, onboarding/activation flow, retention surfaces. It uses web research for current best-in-class patterns and capabilities the operator may not know yet, screenshots of the live app for grounding, and files structured proposals: {angle, observation, evidence/reference, proposed change, expected impact, effort}.
23. Proposals land in a STUDIO INBOX on the studio page (not the general approval queue): swipe-style accept (→ becomes a studio task, variants included) / reject / "later". Accept/reject feeds Part 5 preference learning so scouts learn what the operator values. Cap `ORCH_UX_SCOUT_OPEN_MAX` (default 10 open proposals) — scouts pause when the inbox is full; stale proposals expire in 14 days.
24. Scout quality loop: track each accepted proposal's downstream outcome (persona scores Part 7, and post-promote product KPIs where available); scouts' angle rotation is weighted toward angles whose proposals get accepted AND perform — the scout gets better at scouting.

## PART 7 — PERSONA QA / PROTOTYPE BOTS (demand-guided variant selection)

25. `market_personas` table: target user groups per project — demographics, goals, device mix, tech comfort, context of use. THREE sources, merged with provenance:
    a. AUTO-CURATED (default, runs first): `runner/persona_curator.py` deeply reviews each app's code and product surface — routes/pages, copy and value propositions, pricing/signup flows, feature set, data models, integrations — plus web research on the category and comparable products, and infers the expected/likely target market groups (primary, secondary, anti-personas), each with reasoning and confidence. Presented in the studio UI for operator review/edit/approve; re-runs when an app's product surface changes materially (new routes/pricing/features detected via merged diffs).
    b. OPERATOR-DEFINED: manual definitions/overrides always win over inferred ones.
    c. SIGNAL-ENRICHED: analytics exports, reviews, and support tickets dropped into intake refine both of the above; discrepancies between inferred and observed users are flagged ("code targets SMB owners; analytics show freelancers") as strategic findings, not silently merged.
    Extend the existing persona machinery (`cade_persona_synthesis`, `cade_owner_preference_simulator`) to synthesize rich persona profiles from all three sources.
26. PERSONA PANEL EVALUATION: when variants are ready (Part 2), a QA-bot pass evaluates each variant per persona — multimodal review of the rendered screenshots/flows scoring desirability, clarity, trust, perceived value, friction (1–10 with reasoning), plus task-completion walkthroughs via Playwright (can the "low tech comfort, mobile" persona complete signup on this variant?). Aggregate into a per-variant scorecard by user group, displayed on the variant grid next to cost.
27. Log persona predictions vs eventual operator choice and vs real post-promote metrics to calibrate the panel; flag personas whose predictions consistently diverge from reality.

## PART 10 — SHIP CONSOLE: BATCH PUSH, CHANGELOG, AND EASY REVERT (per project, on the studio page)

43. CHANGELOG LEDGER: every change that lands on a project's design branch (studio task, variant pick, auto-applied preference change, brand-kit activation, scout-accepted improvement) appends a `design_changelog` row: commit SHA(s), one-line plain-English summary (generated at merge time by a cheap model), intent_id, source (operator prompt | scout | auto-apply | kit), before/after screenshots where UI-visible, cost, timestamp. The studio page shows this as an easy-to-read accumulated-changes list per project tab — this is the batch of unshipped work, auto-saved by construction (it's all committed to design).
44. CLICK-TO-EXPLAIN: clicking a changelog entry opens an inline terminal-style panel showing the actual diff with syntax highlighting PLUS the plain-English explanation of what changed and why, side-by-side with its before/after screenshots. No git knowledge required to understand any entry.
45. PUSH ALL: one button per project tab = `promote(project)` for the accumulated batch — design → master through the full merge train + gates → prod deploy. Shows a pre-flight summary first (N changes, plain-English list, persona scores, est. risk) with confirm. Also supports PUSH SELECTED: check a subset of changelog entries; unselected changes are temporarily rebased out (cherry-pick the selected set onto a promote branch), so partial batches ship cleanly while the rest keep accumulating.
46. EASY REVERT, FULL OR PART: every changelog entry has a Revert button — generates `git revert` of that entry's commits on design (and, if already promoted, offers a follow-up revert-promote to master with the same one-click flow). REVERT ALL resets design to master (with confirm + the list of what's being discarded). Reverts are themselves changelog entries (revertible). If a later change conflicts with reverting an earlier one, show the conflict in plain English and offer an agent-assisted untangle task instead of failing silently.
47. SAFETY: promote/push/revert-to-master actions are operator-only (auth-gated), always through the merge train gates, and logged with intent_id. Nothing in the ship console can bypass the deploy guard.
