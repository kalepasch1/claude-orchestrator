## 1. CADE Precedent Reuse & Evolution (kernel; consumed by Tomorrow war/negotiation rooms + Apparently + Smarter)

Goal: never re-run a determination we've effectively already made. Reuse prior CADE outcomes when still valid; evolve them when inputs shift; only run fresh when genuinely novel.

- **P-1 Determination memo store + semantic index** — persist every `Determination` with an **issue fingerprint**: embedding of issue text + structured key (clause family, jurisdiction, party-position vector, materiality, roster class) + the input snapshot it was decided on. Indexed in pgvector (Apparently corpus infra) / Tomorrow proof store. 
- **P-2 Validity predicate (the reuse gate)** — a cached determination is reusable ONLY if: (a) semantic + structured match above threshold; AND (b) **no invalidating change** since it was decided — checked against the X2 regime oracle (no rule/reg change on that clause-family × jurisdiction), the market-data layer (no pricing regime shift for hedgeable issues), the clause-marketplace acceptance curve (no drift beyond epsilon), and the calibration flywheel (persona reliabilities haven't materially moved). Any miss → not reusable. Fail-closed: unknown validity = re-run. Proof: reuse-gate test where a fixture reg change invalidates an otherwise-matching cached determination.
- **P-3 Three-tier resolution ladder** — on each issue: (1) **Exact/valid reuse** → return cached determination + proof, near-zero cost, badge "reaffirmed (decided <date>, still valid)"; (2) **Warm evolve** → cached determination seeds an incremental re-determination (only changed sub-issues re-run via the recursive sub-question mechanism; unchanged factions/authorities carried forward with their prior positions) — big cost cut vs cold run; (3) **Cold run** → full panel only when novel or invalidated. Track hit-rate per tier; target rising reuse over time. Proof: ladder routing test across the three cases.
- **P-4 Precedent lineage + monotonic improvement** — every evolved determination links to its parent; the chain is the auditable "how our position developed." Never *regress* a determination silently: if an evolve produces a weaker/contradictory outcome vs a still-valid parent, flag for review rather than auto-adopt (guards against corpus poisoning / drift). Proof: regression-flag test.
- **P-5 Cross-room compounding (Tomorrow)** — similar war-room disputes explicitly share the store: the 500th indemnity-cap negotiation opens pre-resolved from the accumulated line of the prior 499 (subject to P-2 validity + counterparty-specific overlays from scouting). This is the moat — our rooms get faster, cheaper, and sharper with volume while competitors re-argue from scratch every time. Metric: per-clause-family reuse rate + determination cost trend (must fall with volume).
- **P-6 Portfolio-shared where lawful, siloed where not** — reuse pool is shared across Tomorrow/Apparently/Smarter for *generic legal/market determinations* (no client facts). Client-fact-specific determinations stay within the consent/barrier spine (v4 A-5) — never leak one client's negotiated positions into another's room. Proof: barrier test — client-specific determination never surfaces cross-client.

## 2. Cross-app coordination — scoped by actual overlap (revises v4 §B per operator)

Not every app shares a user or a data model; forcing coordination where there's no overlap adds attack surface for zero value. Scope it:

- **Financial/Legal/Compliance cluster (HIGH overlap — full coordination)**: Tomorrow, Apparently, Smarter, Pareto. Apply v4 B-1 onboarding graph, B-2 wallet, B-3 persona registry, B-4 event fabric, and §1 CADE reuse **fully** across these four. This is where "one entity, done once, everywhere" lives.
- **Consumer/entertainment cluster (LOW overlap with the above)**: Galop, Hisanta. Do NOT force shared onboarding/persona/CADE with the financial cluster. They DO share infrastructure primitives only: the kernel receipt/passport/verifier, the Autonomy Console (X1), the guarantee engine (C-2), public proof pages (C-3), billing rails (B-2 as an *optional* shared wallet, not forced). No shared user graph, no shared CADE pool. Proof: architecture test asserts no consumer-app data path into financial-cluster CADE/user graph.
- **Bridge, only where a real user journey exists**: e.g., a Pareto user funding a child's Hisanta rewards (P5v2 already contemplates this) is an explicit, consented, narrow bridge — not general coordination. Enumerate bridges explicitly; default is isolation. Proof: bridge-allowlist test.
- **B-5 Coordination value gate** — before any new cross-app link ships, require a stated user-journey justification + measured overlap; the allocator (G13) deprioritizes coordination work with no overlap. Prevents speculative integration debt.

## 3. Tomorrow — self-improving contract generation engine (new)

Make instrument/contract creation self-evolving, self-optimizing, self-novel — beyond B9 composite-payoff-compiler and B11 discovery loop (extend, don't duplicate).

- **CG-1 Generative structuring from outcomes** — engine proposes NOVEL instrument/clause structures by learning from realized outcomes across the book: which payoff shapes, triggers, and clause combinations actually delivered the intended risk transfer at lowest cost/dispute rate. Candidate generation → CADE panel review (full expert roster) → backtest (B11) → fail-closed allowlist gate. Nothing novel goes live without panel + backtest + allowlist. Proof: generator emits a candidate never in the seed set; failing candidate never promoted.
- **CG-2 Structure fitness + evolutionary search** — maintain a fitness score per structure (risk-transfer efficiency, funding cost, dispute/breach rate, client comprehension, regulatory robustness); run guided evolutionary search (mutate/recombine high-fitness structures) within the allowlist envelope. The catalog improves itself weekly. Proof: fitness improves over N generations on a fixture book.
- **CG-3 Anti-fragility & adversarial self-play** — a red-team engine continuously attacks the live catalog (find the arbitrage that breaks a structure, the fact pattern that voids a clause, the regime change that inverts a payoff); discovered weaknesses auto-open remediation determinations (CADE) and patch the structure or quarantine it. The catalog hardens against its own worst case before a counterparty finds it. Proof: injected exploit is detected + structure quarantined.
- **CG-4 Comprehension co-optimization** — every generated structure carries a plain-language explanation (W2d card standard) and is scored on client-comprehension; a structure that's brilliant but unexplainable is down-ranked. Keeps "simple, one-click" intact as the catalog grows exotic. Proof: comprehension score gates catalog rank.
- **CG-5 Novelty provenance + defensibility** — every novel structure gets a signed provenance record (what it evolved from, why, panel + backtest evidence) — both an IP/defensibility asset and the audit trail regulators/counsel will want. Proof: provenance verifies; unprovenanced structure cannot enter catalog.
- **CG-6 Governance floor** — self-improvement operates strictly inside: ECP/swap-only/bilateral constitution (locked), the allowlist, materiality gates, and MATERIAL operator post-hoc review for any structurally-new instrument class. Self-novel ≠ ungoverned. Proof: constitution + allowlist deny tests on out-of-envelope candidates.

## 4. Smarter — further junior-associate optimization (new, additive to S1v2/S2v2/S3/S4v2/SC1)

- **SM-1 Autonomous matter project management** — every matter is a live plan: tasks, dependencies, deadlines (A6 engine reused), owners; the system sequences work, drafts each step, and surfaces only human-required nodes (S2v2 flagged-only). Court/filing/SOL deadlines are dead-man-switched. Proof: fixture matter auto-plans + escalates a missed node.
- **SM-2 Communication autopilot with privilege guard** — drafts all routine internal + client updates, status memos, and scheduling; external/client sends stay human-approved (C1 gate); privilege/UPL classifier on every outbound; "you have 3 things to approve" replaces an inbox of 200. Proof: no external send bypasses approval; privilege classifier test.
- **SM-3 Relationship & wellbeing signals (associate-owned)** — associate's own dashboard: workload trend, burnout/overload early-warning from work-pattern telemetry (self-directed, NOT surveillance — mirrors S4v2's aggregate-to-partner boundary), mentorship prompts, matter-anniversary and business-development nudges. Aggregate-only to partners. Proof: named-individual data stays associate-side (access test).
- **SM-4 Career development engine** — skill-gap analysis from matter history, targeted stretch-assignment suggestions, portable verified skill passport (S5), and the recruiting-exchange opt-in — the associate's growth is instrumented and owned by them. Proof: passport claims mint from matter outcomes.
- **SM-5 Autonomous work-product QA** — shadow-associate (S2v2) doubles as a continuous QA layer: every human draft gets a silent CADE-panel review for missed issues, citation errors, deviation from playbook/precedent; surfaced as optional suggestions, never blocking. Proof: fixture draft with a planted error surfaces the flag.
- **SM-6 Knowledge compounding** — SM-1..SM-5 exhaust feeds S1v2 playbooks + the §1 CADE reuse pool (generic determinations only, barrier-guarded). Every matter makes the next one faster. Proof: reuse-pool contribution test with barrier check.

## 5. Apparently — pre-launch proving harness (new; run NOW, before mass onboarding)

Prove and harden every process (drafting/filing/approvals/compliance updates) before real users arrive.

- **AP-1 Synthetic filing lifecycle harness** — fixtures spanning license types × jurisdictions × sectors run end-to-end through the real pipeline (requirements graph → draft → completeness → approval → filing → renewal → exam) against sandboxed/mocked regulator endpoints. Assert: correctness, completeness (no missing element/deadline), determinism, fail-closed behavior. Proof: harness green across the fixture matrix; any gap fails CI.
- **AP-2 Golden-filing regression corpus** — historical successful filings as golden outputs; regenerate through the pipeline and diff; drift fails CI (mirrors CADE golden suite §3.2). Seed ≥20 per major license type. 
- **AP-3 Adversarial/edge fuzzing** — malformed inputs, conflicting requirements, mid-process reg changes, deadline collisions, partial data, revoked consent — assert the system degrades safely (escalates, never mis-files, never over-discloses per R2a). Proof: fuzz suite; zero unsafe outcomes.
- **AP-4 Regulator-perspective validation** — run the R2 data room + empowerment layer (R2f–R2k) against a CADE "skeptical examiner" persona panel that probes for gaps, over-disclosure, unsupported claims, and check-the-box misfits; findings auto-open improvement tasks. Proof: examiner-panel run produces graded report.

## 6. Orchestrator — Persona Test-Bot Swarm (new; continuous, cross-app)

Autonomous synthetic users that live in the products, stress every feature, demand improvements, and safely try to break things.

- **TB-1 Persona fleet** — 5–10 curated personas per app modeling real user archetypes (e.g., Apparently: multi-state MSB compliance officer, first-time gaming licensee, overloaded fintech GC; Tomorrow: regional-bank CRO, hedge-fund ECP, corporate treasurer; Smarter: overworked 2nd-year associate, solo practitioner; Pareto: dual-income family, near-retiree; Galop: casual free-play user, serious handicapper; Triage: frustrated ICU nurse, department chair). Each persona = goals, constraints, sophistication, patience, adversarial tendency.
- **TB-2 Real-usage exercise** — bots drive the actual product surfaces (via the app APIs / test harness, sandboxed data only) end-to-end, scoring: time-to-first-value (v4 A-3), friction points, decision-budget adherence, value delivered, empty states (v4 D-2), comprehension (v4 D-3), guarantee/receipt correctness. Output: per-persona UX + value scorecard.
- **TB-3 Demand-improvements loop** — bots file structured improvement demands (RICE-scored, like real intake) when value/UX falls short; these enter the normal backlog through prompt_factory → auto-decomposed. The fleet critiques itself into improvement. Proof: a fixture shortfall produces a queued improvement task.
- **TB-4 Safe red-team** — bots attempt to break/exploit SAFELY within a hard sandbox: consent/barrier evasion, over-disclosure, guarantee-gaming, prompt-injection of the CADE invokers, RLS/authz probing, money-path abuse (simulated only), anonymity-firewall attacks (Triage). Any success = P0 incident + permanent CI guard (G11 incident-to-guard). HARD LIMITS: sandboxed data only, no production money movement, no real filings, no real external comms, no real user data — enforced by the constitution (deny `testbot_touches_production`). Proof: sandbox-escape attempt is itself blocked + logged.
- **TB-5 Weekly cadence + trend** — scheduled run every ~week per app; track scorecards over time (regressions flagged, improvements verified); feed the daily/weekly fleet brief with a "synthetic-user health" section. Proof: scheduled task registered; trend persisted.
- **TB-6 Value-delivery attestation** — each run emits a signed report: what worked, what didn't, what was demanded, what was exploitable. This is the pre-launch (and ongoing) evidence that the products actually deliver — and the input to Guarantee Everything (C-2) calibration before any guarantee goes live. Proof: signed report verifies.

Sequencing: AP-1..AP-4 + TB-1..TB-4 run BEFORE Apparently mass-onboarding (they are the go/no-go gate); §1 CADE reuse ships early (it cheapens everything after); CG-* and SM-* by G13 allocation; TB-5 weekly cadence turns on once TB-1..TB-4 are green.
